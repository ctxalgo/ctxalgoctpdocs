r500_sync_external_positions¶

[en] Title: Synchronize positions from external trading account. This script is used in the following scenario: You have some CTX strategies trading in the same account, and for some reason, you also performed some manual trading outside of CTX, but on the same trading account. Now you want to inject those positions you traded from outside into those CTX strategies. Those outside positions are called dangling positions because they do not belong to any of the CTX strategies.

You should invoke this script after all the CTX strategies terminate. [zh] Title: 同步外部交易带来的持仓 本程序负责把外部交易产生的持仓植入到CTX的策略中去。它应用于以下场景：你有一些CTX策略，它们使用同一个交易账户。由于某种 原因，你使用外部的方式进行了一些交易，产生了一些持仓，你想把这些外部的持仓植入到CTX策略中去。这些外部产生的持仓叫做 悬挂支持，因外它们不属于任何CTX策略。

你需要在相关的CTX策略都停止后才能执行该脚本。

class ctxalgoctp.ctp.live.r500_sync_external_positions.SynchronizeExternalPositions(instrument_ids, parameters, base_folder, periods=None, description=None, logger=None, strategy_callbacks=None)¶

Bases: ctxalgoctp.ctp.strategy.AbstractStrategy

This strategy’s sole purpose is to get the account’s positions. Those position information is ready after strategy initialization, so we can set the strategy to terminate in the on_before_run method.

Initialize current. :param instrument_ids: [string], the ids of the futures to trade. :param parameters: dict from string to objects. This dict defines the list of parameters for the strategy,

keys are parameter names, and values are the corresponding parameter. Some of the predefined parameter names are:

close_position_time_delta: int, the number of minutes before market close, used to close all
open positions for an intraday trading strategy. For a non-intraday trading strategy, no need to specify this parameter. No need to provide this in a multi-day strategy.
open_position_time_delta: int, the number of minutes before market close, only before this
time, you can place an open order. No need to provide this in a multi-day strategy.
Parameters:
  • base_folder – string, the folder to store log files.
  • periods – [Periodicity], the list of periodicities for which ohlcs are needed.
  • description –

    dict from string to string. Keys are description labels, values are the the content of those description labels. There should be three labels:

    ‘name’: name of the strategy. The strategy names should be unique per CTX deployment. ‘product’: name of the product. ‘description’: the description of the strategy. ‘url_indicators’: the indicator used to construct urls for strategy debugging.
  • logger – LoggerHandler (from ctxalgolib.loggers.logger_handlers). Logger to record strategy execution information. This can be a CompositeLoggerHandler, which contains multiple loggers. For example, a console logger, a file logger and a zeromq logger.
  • strategy_callbacks – AbstractStrategyCallbacks, if not None, provides external strategy callbacks which will be invoked when strategy level callbacks happen. It is useful to attach new behavior to the existing strategy code without change the code itself.

Setup an action, which will be called when all required ohlc bars have arrived. For example, if period=Periodicity.HOURLY, and instrument_ids are the list of all the traded instruments, then action will be called the next time when all hourly bars for the traded instruments have arrived. :param action: function pointer with the following four parameters:

ohlc_kind: string, the ohlc kind, can be ‘time-based’, ‘volatility-based’. period: Periodicity in case of time-based ohlc, or int/float as volatility threshold

in case of volatility-based ohlc.

name: string, the bar arrival action name, same as the name parameter specified here. ohlcs: dict{string: OHLC}, the list of ohlcs, keys are instrument ids, values are the ohlc objects.

Parameters:
  • period – Periodicity in case of time-based ohlc, or int/float as volatility threshold in case of volatility-based ohlc. This is the period for the required ohlc bars. If None, use self.default_period(). This will be passed into the action when it is invoked.
  • instrument_ids – [string], the list of instruments whose last ohlc bar of required period must arrived before action can be invoked. If None, defaults to self.instrument_ids(), which are the full list of traded instruments.
  • ohlc_kind – string, the ohlc kind, can be ‘time-based’, ‘volatility-based’. This will be passed into the action when it is invoked. If None ,use self.default_period_kind().
  • name – string, the bar arrival action name, used to identify different bar arrival requests. This will be passed into the action when it is invoked. If None, a default name will be given.

:return the name for the bar arrival event.

change_position_to(position, price=None, cross_book=True, tick_delta=None, reason='', instrument_id=None, cancellation=None, cancel_in_time=None, market=False)¶

Change the position of the given instrument to new_position. If price is given, all underlying orders are limit orders at that price, otherwise, market orders. :param position: int, the new position for the instrument.

A positive integer means long positions, a negative integer means short positions.
Parameters:
  • price – float, if None, newly placed orders are market orders, otherwise, limit orders.
  • cross_book –

    bool, If True, when placing a limit order, and when order_book is given, place an order cross the order book, meaning:

    If open long or close short, the limit order price will be max(price, order_book[‘ask_price1’]. If close long or open short, the limit order price will be min(price, order_book[‘bid_price1’].

    If price is None and cross_book is False, then place market order(s). If the instrument does not support market order, self.market_order_simulator is used to simulate a market order.

    Note that limit orders crossing order book and simulated market orders need to have order book information. If there is no order book information (no tick has arrived for ths instrument), those orders will be ignored. In such as case, the result is that some requests are ignored. Later when you check your holding position with your expectation, you will notice the difference. Then you can use another change_position_to to place orders again, hoping that between then and now, some tick data for the instrument has arrived, so orders can be placed correctly.

  • tick_delta – int, when crossing the book, the number of ticks to cross. If None, the default value in self.market_order_simulator will be used.
  • reason – string, the reason for this operation.
  • instrument_id – string, instrument id. If None, use self.instrument_id.
  • cancellation – AbstractOrderCancellation, the cancellation policy for current order. If None, no cancellation policy is specified.
  • cancel_in_time – timedelta, the time the order is allowed to be executed before it is cancelled. This is a syntax sugar for setting up TimeBasedCancellation. If not None, you should not specify cancellation as well.
  • market – bool, if True, send market orders.
Returns:

[dict{string:object}], the list of orders.

Each element is a dict containing the following keys: order_ref: string, the value is string, the order reference.

The results are handles to the placed orders, and you can pass one or more of them to self.cancel_orders to cancel those orders.
